A few design points worth knowing, because they reflect decisions rather than defaults:
The converter reads the 44 columns from the workbook header, never hardcodes them. This is deliberate: if a run emits a different column set, that surfaces as a conformance failure instead of being silently masked by a stale constant in the script. The schema stays defined in the dictionary; the converter just tracks whatever was actually emitted.
The round-trip equality is checked at the JSON layer, not the openpyxl layer. That's the correct notion of "same" — §8 defines the canonical structure as the data, free of workbook styling (fills, column widths, fonts). Two workbooks that differ only in formatting but carry identical data are equivalent, and comparing the JSON projections is what expresses that.
The validate command is the API gate in miniature. It enforces exactly what the orchestrator will enforce before admitting a document to reconciliation: nine keys present, evidence rows rectangular, instrument_id/pair_id null at extraction (id-free discipline), and M+K==N with J==0. On AF2026 it read your real N=12/M=10/K=2 and confirmed the invariant. When you move to the API, this function is the conformance check the orchestration code calls on each document's JSON output — same code, different caller.
Two honest limitations, so you're not surprised later. First, the J count (rows extracted from passages not in the LOCATE inventory) is inferred from blank inventory entry cells in the reconciliation sheet; if your reconciliation sheet ever represents "found outside inventory" differently, the J detection needs to match that convention — it's the one place the parser depends on a sheet-authoring habit rather than pure structure. Second, the converter assumes the audit sheets follow the column headers your AF2026 file uses; a differently-headed audit sheet from an older run would convert but with different keys. Both are consequences of the audits being semi-structured — the evidence table is rigid, the audits are prose-ish — and neither affects the evidence rows or the invariant check, which are the load-bearing parts.
